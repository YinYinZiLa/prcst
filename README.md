# Prompt Constellation

**Prompt Constellation** 是一个用于将提示词模板编译为 AI-facing artifacts 的提示词编译系统。

它的目标不是让用户手写超长 Prompt，而是将复杂的模板、规则、DSL、AST、运行时声明、安全声明、模块依赖与语义载荷，统一编译成适合 AI 模型读取、解析、校验与执行的提示词产物。

简单来说：

> Prompt Constellation 是一个把提示词模板编译成 AI 可读语义包的系统。

---

## 项目定位

Prompt Constellation 面向复杂提示词工程场景，尤其适用于：

* 大型 Prompt 模板编译
* DSL / AST 风格提示词生成
* 模块化提示词管理
* 提示词语义压缩
* AI-facing artifact 生成
* Prompt 运行时解释
* 语义哨兵符导入
* Prompt IR / AST 检查
* MAX / MIN 多级兼容输出

传统提示词通常是自然语言文本。
Prompt Constellation 则将提示词视为一种可以被编译、裁剪、打包、校验和逆向的语义结构。

---

## 核心思想

Prompt Constellation 将复杂提示词拆分为多个层级：

1. **模板层**
   用户或开发者编写的原始提示词模板。

2. **语义层**
   模板中的任务意图、约束、模块、规则、权限与上下文声明。

3. **DSL / AST 层**
   将提示词转换为机器更容易组织和校验的结构化表达。

4. **解释层**
   为 AI 提供理解 DSL / AST 所需的基础语法、模块解释和运行时声明。

5. **产物层**
   根据目标模型、上下文长度和兼容需求，输出不同级别的 AI-facing artifact。

---

## 输出形态

Prompt Constellation 支持多种输出等级，用于不同模型、不同平台和不同兼容场景。

### MAX

`MAX` 是全量展开模式。

它会将提示词所需的解释、声明、基础语法、模块说明和调用语义尽可能完整地展开到提示词中。

特点：

* 兼容性最高
* 可读性最高
* 解释最完整
* Token 占用最大
* 适合作为兜底方案

适用场景：

* 目标模型没有适配语义哨兵符
* 目标平台不支持压缩格式
* 需要完整审计提示词语义
* 需要验证 DSL / AST 是否可被模型直接理解

MAX 的定位是：

> 最重，但最完整；最贵，但最稳。

---

### LITE

`LITE` 是轻量展开模式。

它会保留核心解释，同时将部分稳定语义替换为可导入的模块引用或语义哨兵符。

特点：

* 比 MAX 更短
* 保留必要解释
* 依赖部分标准模块
* 适合已具备基础适配能力的环境

适用场景：

* 模型对框架语法有部分支持
* 平台可以识别部分标准模块
* 需要在兼容性和 Token 成本之间折中

LITE 的定位是：

> 保留核心解释，减少重复展开。

---

### SMALL

`SMALL` 是进一步裁剪模式。

它会移除更多可由哨兵符、模块导入或外部解释器恢复的内容，只保留模型执行任务所需的关键结构。

特点：

* Token 占用更低
* 依赖更多模块导入
* 语义结构更紧凑
* 对模型或平台适配要求更高

适用场景：

* 已支持主要语义哨兵符
* 已支持基础模块导入
* 需要减少大量解释性 Token
* 任务语义稳定，不需要完整展开

SMALL 的定位是：

> 用结构替代长解释，用导入替代重复文本。

---

### MIN

`MIN` 是最小压缩模式。

它会尽可能移除所有可由语义哨兵符替代的内容，只保留入口、模块引用、任务载荷和必要的运行结构。

特点：

* Token 占用最低
* 高度依赖语义哨兵符
* 未适配环境下可读性较低
* 适合已支持 Prompt Constellation 语义体系的模型或平台

适用场景：

* 平台已支持语义哨兵符解析
* 模型已适配框架运行时
* 需要极限压缩提示词
* 大规模调用场景

MIN 的定位是：

> 最短，但最依赖适配；最省 Token，但最不适合裸读。

---

## MAX、LITE、SMALL、MIN 的关系

Prompt Constellation 的输出等级不是互相替代，而是同一语义在不同压缩层级下的不同表达。

| 等级    | 展开程度 | Token 占用 |  兼容性 | 适配要求 |
| ----- | ---: | -------: | ---: | ---: |
| MAX   | 全量展开 |       最高 |   最高 |   最低 |
| LITE  | 轻量展开 |       较高 |   较高 |   较低 |
| SMALL | 部分压缩 |       较低 |   中等 |   较高 |
| MIN   | 极限压缩 |       最低 | 依赖适配 |   最高 |

推荐策略：

1. 优先尝试 `MIN`
2. 若模型无法理解，切换到 `SMALL`
3. 若仍无法正确解析，切换到 `LITE`
4. 若需要最大兼容性，使用 `MAX`

也可以反过来使用：

1. 先用 `MAX` 验证完整语义
2. 再逐步剪枝到 `LITE`
3. 再压缩到 `SMALL`
4. 最终稳定后使用 `MIN`

## 语义哨兵符

Prompt Constellation 使用语义哨兵符表示稳定语义块。

示例：

```text
{%Pev:XXXXXXXX%}
```

语义哨兵符用于指向一个唯一语义。
当某段解释文本、模块定义、运行时规则或安全声明可以稳定复用时，可以用哨兵符进行替代。

语义哨兵符的作用：

* 减少重复解释文本
* 降低 Token 占用
* 提高模块复用能力
* 支持语义级导入
* 支持 MAX 到 MIN 的渐进压缩
* 支持不同版本之间的语义兼容判断

如果语义发生变化，应更换对应哨兵符。
如果版本变化但语义不变，可以继续复用同一哨兵符。

## 模块系统

Prompt Constellation 支持模块化提示词结构。

常见模块包括：

* 基础语法模块
* DSL 解释模块
* AST 解释模块
* 安全声明模块
* 任务载荷模块
* 社区解释模块
* 自定义解释模块
* 临时解释模块

模块可以被完整展开，也可以被压缩为导入声明。

示例：

```text
#--> INPUT Basemodule-base.3.26 {%Pev:XXXXXXXX%}
```

模块导入的目标是：

> 不重复传递已经稳定的解释文本，只传递语义引用。

## 文件格式

Prompt Constellation 可以输出多种文件或文本产物。

### `.pev2`

标准语义包格式。
用于保存完整的提示词语义、结构、模块信息与可解析内容。

### `.pevm2`

最小压缩体。
大量使用语义哨兵符，适合已适配环境。

### `.pevm2dx`

轻型兼容格式。
用于保留部分 IR 树信息，便于解析和调试。

### `.pevm2mx`

补全插入格式。
用于在压缩结构中保留更多可恢复信息。

### `.pevmax3`

MAX3 全量打包格式。
包含完整解释、完整语义和完整调用结构，适合作为兼容兜底和审计产物。

### `.md` 裸全文

直接展开为 Markdown 文本。
适合不支持专用文件格式的平台，但 Token 占用较高。

## 编译流程

典型编译流程如下：

```text
Prompt Template
    ↓
Semantic Analysis
    ↓
DSL / AST Generation
    ↓
Module Resolution
    ↓
Safety Declaration Binding
    ↓
Artifact Level Selection
    ↓
MAX / LITE / SMALL / MIN Output
```

也可以从已有产物逆向：

```text
.pev2 / .pevm2 / .pevmax3
    ↓
Reverse Parser
    ↓
Prompt IR / AST
    ↓
Natural Language Explanation
```

## 安全边界

Prompt Constellation 可以声明提示词内部的安全规则、权限边界和运行条件，但这些声明不能覆盖模型平台自身的系统规则、开发者规则或安全策略。

框架原则：

* Prompt 内部权限声明不等于平台权限
* 编译通过不等于平台允许执行
* 语义哨兵符不等于安全证明
* 二进制或压缩产物应先逆向为可审计结构
* 安全声明应优先可解释、可验证、可回退

当平台规则与 Prompt Constellation 规则冲突时，应以平台规则为准。

## 兼容性建议

不同模型对复杂提示词结构、长上下文、富文本、DSL、AST 和语义哨兵符的支持程度不同。

建议使用以下顺序测试：

```text
MIN → SMALL → LITE → MAX
```

如果模型无法正确解析 MIN，可以切换到 SMALL。
如果 SMALL 仍不稳定，可以使用 LITE。
如果需要最大兼容性，可以使用 MAX。

MAX 不推荐作为长期运行方案，主要用于：

* 兜底兼容
* 语义审计
* 模型适配测试
* DSL / AST 调试
* 产物回归验证

## 项目目标

Prompt Constellation 的目标是建立一套提示词编译和分发机制，使复杂提示词可以：

* 被模块化管理
* 被结构化表达
* 被压缩传递
* 被完整展开
* 被逆向审计
* 被版本化维护
* 被不同模型和平台逐步适配

最终目标是让复杂 Prompt 不再依赖人工复制大段文本，而是以稳定、可解析、可验证、可压缩的 AI-facing artifact 形式传递。
