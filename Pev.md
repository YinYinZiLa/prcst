# Pev

**Pev 是面向 AI 代码生成的意图规格系统。**

它用于把业务逻辑表达为可版本化、可再生成、可跨语言实现的稳定逻辑骨架，使代码不再是业务意图的唯一事实来源，而是由 Pev 生成出来的工程产物。

简而言之：

```text
Pev = intent spec
Prompt = transport format
Code = generated artifact
```

---

## 1. Pev 的核心定义

Pev 是一种面向 AI 的代码生成规格系统，用于描述代码背后的业务意图、逻辑结构、权限边界、生成规则和版本差异。

它不是普通提示词，也不是传统 PRD。

Pev 的目标是让同一份业务意图在不同时间、不同模型、不同目标语言下，仍然能够稳定生成同一逻辑骨架的代码。

```text
Pev v1 → Code v1
Pev v2 → Code v2
diff(Pev v1, Pev v2) → 解释 Code diff 的业务原因
```

在 Pev 管辖范围内，代码不再是业务逻辑的唯一载体。
Pev 才是代码逻辑的真实来源。

---

## 2. Pev 为什么存在

在现实开发中，业务意图经常被沉淀进代码本身。

随着时间推移，开发者和 AI 往往只能从现有代码中反推需求：

```text
Code → 推测业务意图 → 修改 Code
```

这会带来几个问题：

* 代码结构可能已经脏了；
* 历史实现可能并不等于真实业务意图；
* AI 容易过度尊重旧代码；
* 开发者容易陷入“最小改动”思维；
* 业务逻辑变更很难从代码 diff 中直接看懂；
* 跨语言重写时，需要重新理解旧代码和旧架构。

Pev 试图倒置这个关系：

```text
Pev → 生成 Code
```

也就是说，代码是 Pev 的生成结果，而不是 Pev 的来源。

---

## 3. Pev 的核心目的

Pev 的核心目的不是让 AI 写出更多代码，而是让 AI 在代码生成中稳定复现同一个业务逻辑骨架。

它关注的是：

* 业务意图的显式表达；
* 逻辑结构的稳定复现；
* 代码生成的可版本化；
* 不同语言实现之间的逻辑一致性；
* Pev 版本变更与代码变更之间的可解释关系；
* 让 AI 优先理解 Pev，而不是从历史代码中猜测需求。

Pev 希望实现的是：

```text
同一个 Pev
→ 不同模型
→ 不同时间
→ 不同目标语言
→ 仍然生成逻辑相似、结构稳定、目的统一的代码
```

例如，一个模块原本由 Python 实现，后续需要改为 Rust 实现。
在 Pev 模式下，理想路径不是让 AI 阅读旧 Python 代码后重写 Rust，而是让 AI 基于同一个 Pev 重新生成 Rust 实现。

```text
Pev → Python implementation
Pev → Rust implementation
```

代码语言可以变化，但业务逻辑骨架不应漂移。

---

## 4. Pev 与代码的关系

传统开发中，代码通常被视为事实源：

```text
Business intent → PRD → Code → Maintenance
```

但在长期维护中，PRD 可能过时，代码会变成事实上的业务意图存储。

Pev 改变了这个关系：

```text
Business intent → Pev → Code
```

在 Pev 模式下：

* Pev 绑定代码逻辑；
* 代码不绑定 Pev；
* 代码可以被重新生成；
* 代码结构可以被重构；
* 旧代码不是必须妥协的事实源；
* Pev 才是业务逻辑的版本化载体。

因此，Pev 鼓励开发者和 AI 跳出“只做最小改动”的限制。

如果旧代码结构本身就是错误的、脏的、过时的，Pev 允许 AI 基于真实意图进行更激进的重构，而不是继续围绕历史实现打补丁。

---

## 5. Pev 与 AI Agent

Pev 的目标之一，是让 Agent 从“维护代码”转向“维护意图规格”。

在传统 AI 编程流程中，Agent 通常会：

```text
读取代码
→ 推理需求
→ 修改代码
→ 解释 diff
```

在 Pev 模式中，Agent 应该优先：

```text
读取 Pev
→ 理解业务意图
→ 比较 Pev 版本
→ 生成或重生成代码
→ 用 Pev diff 解释 Code diff
```

也就是说，Agent 不再只是代码修改器，而是 Pev 版本的维护者和代码产物的生成者。

这使得 AI 可以更清楚地理解：

* 为什么这次要改；
* 改动的是哪一层业务逻辑；
* 哪些代码变化是 Pev 版本变化导致的；
* 哪些旧结构可以被废弃；
* 哪些逻辑必须保持稳定。

---

## 6. Pev 的架构定位

Pev 是面向代码生成的稳定复现提示词内核。

它通过提示词形式传递给 LLM，使 LLM 在生成代码前先理解 Pev 所描述的业务意图、逻辑骨架、约束边界和生成规则。

Pev 的基本结构可以理解为：

```text
[最小语法支持模块]
+ [扩展模块]
+ [真实任务]
+ [真实任务扩展]
→ LLM 解释 Pev
→ LLM 按 Pev 协议生成代码
```

其中：

* 最小语法支持模块用于定义 Pev 的基础解释规则；
* 扩展模块用于补充领域规则、生成风格、权限边界和错误处理；
* 真实任务用于描述当前业务目标；
* 真实任务扩展用于承载额外上下文和局部约束。

Pev 本身通过 prompt 作为传输格式进入 LLM，但 Pev 的目标不是成为普通 prompt，而是成为 AI 能够稳定解释的 intent spec。

---

## 7. Pev 的版本意义

Pev 的版本变化应当解释代码变化。

```text
Pev v1 → Code v1
Pev v2 → Code v2
diff(Pev v1, Pev v2) → Code diff 的业务原因
```

这意味着，开发者不必只从代码 diff 里反推业务变更，而可以直接通过 Pev diff 理解这次代码变化的真实目的。

Pev 版本记录的是业务逻辑如何变化。

Code 版本记录的是 Pev 在某种语言、某种框架、某种工程环境下的实现结果。

---

## 8. Pev 与 PRD 的区别

Pev 不是传统 PRD。

PRD 通常面向人类阅读，表达方式偏自然语言，容易模糊、冗长、不可直接生成代码。

Pev 面向 AI 代码生成，要求业务逻辑更结构化、更稳定、更适合被 LLM 解释和复现。

可以理解为：

```text
PRD = 人类协作说明
Pev = AI 代码生成意图规格
Code = Pev 的工程实现产物
```

Pev 允许不懂开发的人更直接地表达业务逻辑，而不是先写传统 PRD，再由开发者或 AI 从 PRD 中二次解释需求。

在更激进的使用方式中，Pev 可以绕过传统 PRD 的重文档流程，直接作为 AI 代码生成和重构的业务意图入口。

---

## 9. Pev 适合解决的问题

Pev 主要适用于稳定业务逻辑的代码生成与再生成场景。

例如：

* 同一模块需要跨语言重写；
* 历史代码结构已经失控，需要按业务意图重构；
* AI 多次生成代码时逻辑容易漂移；
* 开发者希望从业务意图而不是旧代码理解系统；
* 需要解释不同代码版本背后的业务原因；
* 需要让 Agent 维护规格，而不是只维护代码；
* 希望将代码从事实源降级为可再生成产物。
